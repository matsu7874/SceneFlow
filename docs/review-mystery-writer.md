# SceneFlow ミステリー作家レビュー

> 視点: ミステリー（本格・館もの・群像劇）を書く立場から、SceneFlow を「整合性チェック付きの執筆支援ツール」として評価する。
> 対象コミット: `fe97d8a`（branch: `claude/mystery-writer-tool-review-gyiKk`）

---

## 総評

結論から言うと、**ミステリー作家にとって"刺さる"骨格を持ったツール**だ。
群像劇の時系列を「人物・場所・小道具・情報・行動」に分解し、行動を**事前条件／事後条件を持つ関数**として捉える設計（`DESIGN.md` の思想）は、まさにフェアプレイ・ミステリーが要求する「誰が・いつ・どこで・何を知り・何を持っていたか」の検証そのものだ。

実際、付属の学院ミステリーサンプルを検証エンジン（`analyzeStory`）に通すと、作家が仕込んだ破綻を正確に拾う:

```
[item]     act9 : 古い鍵 は 図書室 に無いため取得できません（既に持ち出し済み）
[position] act12: レオ は 廊下 におり 中庭 にいないため「…」ができません（移動の欠落）
[info]     act14: ニコ は「礼拝堂の隠し扉」を知らないため話せません（知らない情報を口にした）
矛盾検出  : ニコは「アランは昏倒」を目撃済みなのに、又聞きの嘘「自室で就寝」を受け取り矛盾
```

この4種——**動線・所持・知識・証言の食い違い**——を自動で突けるのは、プロットの穴探しを手作業でやってきた身からすると素直に価値がある。とりわけ最後の「目撃した真実」と「吹き込まれた嘘」の衝突（`truth-conflict`）を検出し、因果ビューで嘘の伝播経路を二色で遡れるのは、**叙述の信頼性・証言の連鎖を設計する道具**として独創的だ。

以下、作家として「あと一歩」と感じた点を、重要度順に挙げる。

---

## 重要度: 高 — ミステリーの核心が未対応／届いていない

### 1. 密室・施錠・鍵が検証されない（型にあるのに使われていない）

`Location` 型には `properties.isLocked` / `properties.requiredItem`、`Prop` には `isPortable`、`Location` には `capacity` が定義されている。ところが**整合性エンジン（`src/modules/consistency/`）はこれらを一切参照していない**（grep で確認: ヒット0件）。

- `MOVE` は隣接（`connections`）しか見ない。**施錠された礼拝堂の地下に、鍵を持たない人物が入っても警告は出ない。**
- 「密室」「鍵を持つ者だけが入れる部屋」「凶器は持ち出せない大道具」——本格ミステリーの中核トリックが、データ上は書けるのにエンジンは黙認する。

作家の期待は明確だ:「`requiredItem` を所持していなければ `isLocked` な場所に `MOVE` できない」「`isPortable: false` の `銀の燭台` は `TAKE` できない（＝凶器を現場から動かせない）」。この一手が入るだけで、**密室トリックの検証ツール**に化ける。

### 2. 移動に「時間」の概念がなく、アリバイの時間検証ができない

アリバイ崩しはミステリーの華だが、現状の移動判定は**隣接しているか否か**だけ。`Act` 型に `endTime` はあるが未使用で、行動はすべて時刻上の点として扱われる。

- 「図書室から礼拝堂まで5分で移動できたか？」という**所要時間の問い**に答えられない。距離・移動コストの概念がないため、隣接さえしていれば0分で瞬間移動できてしまう。
- 同時刻に別の場所に居る矛盾（`colocation`）は拾えるが、これは「2点に同時存在」という極端なケースのみ。**「物理的に間に合わない移動」**という、アリバイトリックの主戦場は素通りする。

提案: `location` 間に移動コスト（分）を持たせ、連続する `act` の時間差が移動所要時間を下回ったら警告。これでアリバイ表（タイムテーブル）の検算が自動化できる。

### 3. 「事件そのもの」を状態として表現できない

サンプルでも、クライマックスの「アランと対峙する」(act16) や被害者の「昏倒」は**型なしの `act` と自由記述の情報**でしか表現されていない。ワールドステート（`worldState.ts`）が追跡するのは位置・所持・知識・消費の4つのみで、**生死・負傷・意識といった人物の状態が無い**。

被害者の死という、ミステリーで最も重要な「事後条件」がモデル化できないため、「死んだはずの人物がその後行動している」「昏倒中の人物が会話している」といった**致命的破綻を検出できない**。`ATTACK`/`INCAPACITATE` のような状態変化アクトと、人物の状態スロットが欲しい。

### 4. 証言矛盾（`contradictions`）が「検証」ページに出てこない

これは**実装はされているのに導線が惜しい**問題。エンジンは `truth-conflict`/`testimony-conflict`（目撃した真実と吹き込まれた嘘の衝突）を計算しているのに、**検証ページ（`ValidationReporter`）は `breakages` しか表示せず、`contradictions` を捨てている**。矛盾は因果ビューでノードを選択したときだけ見える。

作家は穴探しのとき真っ先に「検証」タブを開く。**そこに「証言の食い違い」サマリが出ない**のは、最大の売りを隠しているに等しい。`contradictions` を検証リストの一カテゴリ（例: `testimony` 重大度 warn）として足すだけで体験が大きく変わる。

---

## 重要度: 中 — 情報・推理まわりの表現力

### 5. 会話は「1対1」——立ち聞き・公言・密談の区別がない

`SPEAK` は `interactedPersonId`（ただ1人）にしか情報を伝播しない。だが現場に第三者が同席していれば、その人物も聞いているはず。

- **立ち聞き（盗み聞き）**＝ミステリーの定番が表現できない。「同じ部屋に居た全員が聞く」「壁越しに隣室の1人だけが聞く」を選べると、情報拡散トリックが組める。
- 逆に「2人きりの密談」を保証したいなら、同席者がいないことを検証したい場面もある。

### 6. 推理・演繹のモデルが無い（知識は"渡される"だけ）

知識は `LEARN`/`SPEAK` で**外から付与される**もので、単調増加（一度知れば忘れない）。これは妥当だが、探偵が複数の手がかりから結論を**導出**する筋が表現できない。「手がかりA＋手がかりB → 真相Cを知る」という `enablesActions`/`revealsInformation`（型にはある）を使った推論連鎖を検証できると、フーダニットの"解決編"まで設計支援が届く。

### 7. 「機会・容疑者」の逆引きがない

現状は**作家が書いた筋書きを検算する**ツール。ミステリー支援としてもう一段欲しいのは逆引きだ:

- 「犯行時刻に現場（礼拝堂）に居られた人物は誰か」
- 「凶器（銀の燭台）に触れられた人物は誰か」
- 「この秘密を知り得た人物は誰か」

これらを一覧する**"容疑者リスト"ビュー**があれば、フェアな手がかり配置と消去法の設計に直結する。データはすべて揃っているので、集計ビューを足すだけで実現可能。

## 重要度: 低 — 細部・堅牢性

- **`relationships.targetId` が string、`person.id` が number** で型が食い違う（`StoryData.ts`）。関係性ダイアグラムでの突き合わせが壊れやすい。id 体系を number に統一すべき。
- **`type` 無しの `act` は「その他」扱い**で位置チェックしか走らない。サンプルの山場（対峙・考え事）がこれに該当し、検証の網から漏れる。重要アクトには種別付けを促す UI ヒントがほしい。
- **消費（`USE` + `isConsumable`）の事後条件は良い**（`鎮静剤` は飲めば消える）が、`combinesWith`/`isCombineable`（道具の組み合わせ）はエンジン未対応。錠剤と毒の調合のようなトリックは書けない。
- **真実スロットの重複検出（`duplicateTruthSlots`）も計算済みだが UI 導線が因果ビュー寄り**。オーサリング警告として検証ページにも出したい。

## ミステリー作家としての評価まとめ

| 観点 | 評価 | コメント |
| --- | --- | --- |
| 動線の整合（移動の連続性） | ◎ | 隣接違反・瞬間移動を確実に検出 |
| 所持の整合（誰が何を持つか） | ◎ | 取得・受渡・設置・消費の前提条件を追跡 |
| 知識の伝播（誰が何を知るか） | ○ | 連鎖は追えるが1対1のみ・推論なし |
| 証言の矛盾（嘘と真実の衝突） | ◎ 着想 / △ 導線 | 独創的だが検証ページに出ない |
| 密室・施錠・鍵 | ✗ | 型はあるが未実装。本格の核心が欠落 |
| アリバイの時間検証 | ✗ | 移動コスト・所要時間の概念が無い |
| 事件（生死・負傷）の状態 | ✗ | 被害者の状態をモデル化できない |
| 容疑者・機会の逆引き | ✗ | 検算はできるが推理支援は未提供 |

**一言で:** 「動線・所持・知識・証言」の検算エンジンとしては既に実用域で、群像劇の破綻チェックに今日から使える。ただし**密室・アリバイ時間・事件の状態**というミステリー三種の神器が未対応なので、現状は「ミステリーも書ける群像劇ツール」。ここを埋めれば「**本格ミステリーの整合性チェッカー**」という独自ジャンルの定番になり得る。

### 作家として優先してほしい改修トップ3

1. **施錠・鍵・可搬性の検証**（`isLocked`/`requiredItem`/`isPortable` をエンジンで使う）= 密室が書ける
2. **移動所要時間の導入**（場所間コスト × 行動の時間差）= アリバイが検証できる
3. **証言矛盾を「検証」ページに表示**（`contradictions` を出す）= 既存の最大の強みを表に出す
