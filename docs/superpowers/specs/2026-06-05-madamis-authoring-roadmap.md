# マダミス作家向け機能ロードマップ（今後のネタ）

- 日付: 2026-06-05
- 対象: SceneFlow（React 19 + TypeScript + Vite）
- 位置づけ: マーダーミステリー（マダミス）／イマーシブシアター脚本制作の観点で実施したツールレビューから抽出した機能アイデア集。すぐ着手する確定仕様ではなく、優先度判断と将来設計の出発点とする。

## 背景

SceneFlow は「誰が・どこで・何を持ち・何を知っているか」の整合性を自動検証できる点で、マダミス制作の中核作業（破綻のないシナリオ構築）に強く適合している。学院ミステリのサンプル（`misinfoType:"lie"`、SECRET/FACT/RUMOR の情報分類、所持・移動・SPEAK/LEARN による情報伝播、意図的に仕込まれた矛盾）が示すとおり、必要なデータモデルは既に揃っている。

一方で「作品を仕上げて配布する」「フェアな謎として成立しているか確かめる」段階を支える機能はまだ薄い。本ドキュメントはその不足を埋める 4 つの機能候補を、既存データ資産を活かす前提でまとめる。

---

## 提案1: キャラクター別ハンドアウト（HO）出力 ★最優先候補

### 課題

マダミスでは各プレイヤーに「そのキャラクターの視点・知っている情報・目的」を記したハンドアウト（HO）を配る。現状の SceneFlow はゲームマスター視点の全体ビューはあるが、プレイヤー配布物を生成できない。

### 提案

人物ごとに以下を含む HO を Markdown / 印刷向け HTML で出力する。

- そのキャラクターのプロフィール（名前・職業・説明・目標）
- そのキャラクターが体験する時系列（自分が主語の Act を時刻順に並べたもの）
- そのキャラクターが「知っている情報」（LEARN / SPEAK で受け取った information を、取得時刻つきで）
- 開始時点の所持品・初期位置

### 既存データとの関係

- 視点フィルタ: `acts[].personId` と `interactedPersonId` で当該人物が関与する Act を抽出。
- 知識の算出: SPEAK/LEARN の Act を時系列に再生し、各時点で人物が保持する `informations` 集合を求める（シミュレーションの実行ログ生成ロジックを流用可能）。
- 所持・位置: `initialStates` と props の owner から導出。

### 期待効果

データ入力済みの作品から、そのまま配布物が出る。マダミス作家にとって「作って終わり」でなく「卓に出せる」ところまで到達する目玉機能。

### 規模感

中。新規ページ／パネル＋知識再生ロジック（既存流用多め）＋出力テンプレート。外部依存は最小（Markdown 文字列生成で足りる）。

## 提案2: 人物の「秘密 / 動機 / アリバイ」構造化フィールド

### 課題

現状、人物の秘密や動機は `description` の自由記述に混在している（サンプルでも「ある秘密を抱える」と本文に埋め込まれている）。構造化されていないため、検証・HO 出力・相関図で機械的に扱えない。

### 提案

Person 型に任意フィールドを追加する。

- `secrets`: 秘密（information への参照、または自由記述の配列）
- `motive`: 動機（被害者・事件に対する動機）
- `alibi`: 主張するアリバイ（時刻範囲＋主張する居場所）。検証エンジンが実際の動線と突き合わせ、「アリバイが事実と矛盾する／成立する」を判定できる。

さらに発展として「制約」を持てると強力:

- 「情報 X を人物 Y に知られてはいけない」という禁止制約。シミュレーション再生中に X が Y に到達したら警告（＝バレてはいけない秘密が露見する瞬間の検出）。

### 既存データとの関係

- `informations` の SECRET と人物を結びつけることで `secrets` を表現可能。
- アリバイ検証は既存の「位置・動線」整合性チェック（consistency エンジン）の延長。

### 期待効果

秘密・動機・アリバイが第一級データになり、検証・HO・相関図すべてで一貫して扱える。提案1・3・4 の土台。

### 規模感

小〜中。型追加＋エンティティ編集フォーム拡張。制約検証まで含めると中。

## 提案3: 可解性（フェアプレイ）チェック

### 課題

現状の検証は「矛盾の検出」（破綻＝やってはいけないことの発見）に特化している。マダミスでは逆に「探偵／プレイヤーが犯人や真相を論理的に特定できるだけの手掛かりが配置されているか」（フェアプレイ）も重要だが、これは支援されていない。

### 提案

真相（犯人・凶器・手口など、作者が指定する「解」）に対して、

- その結論に到達するために必要な手掛かり（information / prop / 目撃 Act）が、ゲーム中にいずれかのプレイヤーから到達可能か
- 逆に、誤った結論を排除できる手掛かりが揃っているか

を判定し、「この手掛かりが欠けると犯人を特定できない」「この情報は誰も入手経路がない（デッドクルー）」を指摘する。

### 既存データとの関係

- 因果関係ビューの依存グラフ（事実・証言の供給元／依存先）を逆向きに辿る到達可能性解析として実装できる。
- 「矛盾発覚点」の概念の自然な拡張。

### 期待効果

「難しすぎて解けない／簡単すぎてネタバレ」を設計段階で検出。フェアな謎を作るための定量フィードバック。

### 規模感

中〜大。作者が「解」を入力する仕組み＋到達可能性／必要十分性の解析が必要。まず「到達不能な情報（誰も入手できない手掛かり）の検出」という小さい部分から始められる。

## 提案4: 嘘の安全圏分析（情報到達の可視化）

### 課題

「矛盾発覚点」は強力だが、作者が知りたいのはしばしば「この嘘はいつ・誰の前で破綻するのか」「それまでは安全か」という時間的・人物的な余白である。

### 提案

特定の嘘（`misinfoType:"lie"` の information）について、

- それを信じている人物の集合と、真実を知っている人物の集合の時間変化
- 両者が同席（同じ場所に居合わせる）または会話（SPEAK）する最初のタイミング＝破綻の起点
- 破綻までの「安全圏」（嘘が保たれる時間帯・場所）

をタイムライン上で可視化する。

### 既存データとの関係

- シミュレーションの位置再生＋情報伝播ログから、「嘘保持者 × 真実保持者」の遭遇を検出する到達可能性解析。学院ミステリの Act20（クラウスがニコに又聞きの嘘を伝える→ニコは昏倒を目撃しており矛盾に気づく）がまさにこのケース。

### 期待効果

「嘘が破綻する瞬間」を演出のクライマックスとして意図的に設計できる。マダミスの山場づくりを直接支援。

### 規模感

中。提案3の到達可能性解析と基盤を共有できる。

---

## まとめ・進め方の提案

依存関係を踏まえると、着手順序は次が自然:

1. **提案2（構造化フィールド）** — 他提案の土台。小さく始められる。
2. **提案1（HO 出力）** — 既存ロジック流用が多く、利用者価値が分かりやすい目玉。
3. **提案4（嘘の安全圏分析）** → **提案3（可解性チェック）** — 到達可能性解析の基盤を共有。提案4の方が範囲を絞りやすい。

いずれも既存のデータモデル（acts / informations / 整合性エンジン / 因果グラフ）を土台にできるため、ゼロからの新規構築は不要。
